# Codex 页面切换卡顿修复设计

## 问题与证据

Codex 页面切换时仍会显示其原生蓝色加载圈，但伴随组件运行时卡顿更明显。A/B 诊断显示，组件自身没有可测 CPU 峰值；然而页面切换期间，组件专属 `codex app-server` 子进程出现约 120 KB 读取和 24 KB 写入，静止对照阶段为零。

根因是 Codex 内部页面切换产生的前台窗口事件被组件当作“应用重新获得焦点”。当前逻辑会强制发布相同窗口快照、重复显示或定位伴随窗口，并立即调用 `account/read` 与 `account/rateLimits/read`。这些工作与 Codex 页面加载并发，放大了卡顿。

## 方案选择

- 方案 A：取消所有前台刷新。实现最简单，但无法满足 Codex 真正恢复前台时及时同步的需求。
- 方案 B：区分真实前台转换与同一窗口的重复前台信号，并对相同布局去重。保留真实恢复刷新，同时消除页面切换额外工作。采用此方案。
- 方案 C：保留现有触发条件，仅延迟刷新。可以减轻部分争用，但无法消除多余 RPC 和 Z 序更新。

## 行为设计

新增独立、可测试的前台活动判定状态：

- 全局前台事件指向当前 Codex 顶层 HWND，且上一次已确认前台 HWND 不是该窗口时，发布一次真实 `Foreground` 转换。
- 相同 Codex HWND 连续收到前台事件时，不发布重复状态，不触发额度刷新或伴随窗口重新显示。
- 前台切换到非 Codex 窗口时，只记录 Codex 已失去前台，不隐藏伴随条；随后返回 Codex 时允许发布一次真实转换。
- 最小化、隐藏后恢复继续立即刷新。
- 60 秒定时刷新和手动刷新保持不变。
- 窗口移动、缩放、全屏、展开/收起和 60 FPS 快速跟随路径不变。

## 结构调整

- `CodexWindowTracker` 保存最后确认的前台 Codex HWND；收到全局前台信号时重新读取系统当前前台 HWND，避免把内部表面句柄误当作顶层窗口。
- 重复前台信号降级为普通快照去重，不再依赖 `Foreground` 的无条件强制发布。
- `CompanionHost` 只对真实 `Foreground`、隐藏恢复或最小化恢复请求即时额度刷新。
- 对未变化的收起按钮边界避免重复 `ShowAt`，保持现有原生快速移动路径。

## 测试与验收

- 同一 Codex HWND 连续前台事件只发布一次真实前台转换。
- 前台转到非 Codex HWND 后再返回，发布一次真实前台转换。
- 内部页面切换样式的重复前台事件不触发额度刷新或布局重排。
- 最小化/隐藏恢复、60 秒定时刷新和手动刷新行为不回归。
- 完整 Release 测试、构建、覆盖安装和实机页面切换 A/B 复测通过后更新 GitHub。
